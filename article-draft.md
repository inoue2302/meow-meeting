# AI時代のキャリア不安を猫4匹の作戦会議で癒すアプリを作った

## はじめに

「AIに仕事奪われるかも...」「転職すべき？」「俺もう終わった？」

AI時代のキャリア不安って、人間に相談しても余計に不安になりがち。だったら猫に聞こう。

そんなコンセプトで「にゃんず進路会議」を作りました。

https://meow-meeting.vercel.app/

## コンセプト

手軽にかわいく、AIに進路を相談できるアプリ。

ただの相談では面白くないので、**性格の違う4匹の猫がリアルタイムで会議をして、結論を出してくれる**という演出を入れました。

### 4匹のキャラクター

| 名前 | 役割 | 性格 |
|------|------|------|
| モチ | 共感担当 | 「つらかったにゃね...」まず受け止める。甘えん坊で泣き虫 |
| カゼ | 行動担当 | 「今すぐ動くにゃ！」背中を押す。猪突猛進のお調子者 |
| スミ | 分析担当 | 「冷静に考えるにゃ」ズバッと言う。毒舌だけど本質を突く |
| トラ | 進行+まとめ | 「落ち着くにゃ」会議を仕切って最後にまとめる。ボス猫 |

猫たちは会議の中で業界風刺も自然に混ぜます。「"AIネイティブ"って言ってる人、去年まで"Web3"って言ってたにゃ」みたいに。ただし結論は必ずポジティブ。風刺は「気づき」のため。落とすためではなく、笑って前を向くためです。

## 体験の流れ

1. **ヒアリング** — トラが1対1で質問（すべて選択式）
2. **猫会議** — 4匹がリアルタイムで議論（ストリーミング表示）
3. **ポカポカ演出** — 猫たちの喧嘩アニメーション
4. **結論** — トラのまとめ + 明日からの作戦3つ

## 自由入力を排除した理由

今回はキャリア相談というセンシティブな内容を扱うので、**自由入力を一切排除**しました。

- ユーザーの入力はすべて**事前に定義した選択肢**のみ
- APIに届くのは「テーマID + 選択肢の文字列」だけ
- サーバー側でもホワイトリスト検証（選択肢以外は400で拒否）

これにより**プロンプトインジェクションが構造的に不可能**になります。安全設計の根幹です。

## 技術的な挑戦: リアルタイムストリーミング

単純に結果を返すだけなら簡単ですが、「猫たちが勝手に会話している」演出を入れたかった。そこで**Vercel AI SDKの構造化ストリーミング**を使いました。

### 構成

```
Next.js 15 (App Router)
├── Server Actions (createStreamableValue)
├── Vercel AI SDK (streamText + Output.object)
├── Anthropic API (Claude Sonnet 4)
└── Zodスキーマ (構造化出力)
```

### ネタばらし: API呼び出しは1回だけ

実は猫たちの会議は、**LLMへのリクエスト1回**で全部生成しています。4匹が交互に喋っているように見えますが、裏側では1つのリクエストで以下の構造化JSONを一括生成し、それをストリームに変換してリアルタイムに見せています。

```json
{
  "messages": [
    { "cat": "トラ", "text": "..." },
    { "cat": "モチ", "text": "..." },
    ...12〜15発言
  ],
  "conclusion": "...",
  "strategies": ["...", "...", "..."],
  "finalWord": "..."
}
```

会話12〜15発言 + 結論 + 作戦3つ + 最後の一言、これが全部1回のAPI呼び出しで返ってきます。トークン消費を抑えたかったのと、複数回のAPI呼び出しで会話の整合性が崩れるリスクを避けるためです。

Vercel AI SDKの `streamText` + `Output.object()` + Zodスキーマで、**型安全な構造化ストリーミング**を実現しています。

### ストリーミング表示のガタつきとの戦い

最初の実装では、テキストが1文字ずつ更新されるたびに吹き出しがリサイズされて**ガタガタ**しました。

これを解消するために：

1. **確定済みメッセージとストリーミング中のメッセージを分離** — 確定済みは固定表示、最後の1件だけリアルタイム更新
2. **`transition-all` + `min-height`** — 吹き出しのリサイズを滑らかに
3. **`scrollIntoView`** — スマホでも確実にスクロール追従

「猫が考えてから喋る」自然な体験になりました。

### Server Actions でAPIエンドポイントを隠す

当初はRoute Handler (`/api/meeting`) で実装していましたが、外部から直接叩けてしまう問題がありました。

Server Actionsに移行することで：

- エンドポイントが外部に露出しない
- `createStreamableValue` + `readStreamableValue` でストリーミングをServer Action内で完結
- バリデーション → ホワイトリスト検証 → レート制限の順序で、不正リクエストではレート制限カウントを消費しない

## LLMの非冪等性を武器にする

同じ選択肢の組み合わせでも、毎回違う会議が展開されます。これはLLMの非冪等性（同じ入力でも出力が変わる）を**バグではなくリプレイ性**として活用しています。

`temperature: 1` に設定して会話のブレ幅を最大化。何度やっても新しい発見がある体験を目指しました。

## レート制限とコスト管理

ネタアプリでもAPIコストは現実問題。Upstash Redisで全体のリクエスト回数を制限しています。

工夫したポイント：
- **開始ボタンの時点でチェック** — ヒアリングを全部やった後に「制限超過」と言われるストレスを回避
- **レート制限画面** — 4匹が寝ているイラストで「猫たちはお休み中にゃ...」と表示。エラーも体験の一部にする

## 開発プロセス

Claude Codeを使って開発しました。特にレビューではtmuxで別セッションからAIレビューを並列実行し、must/imo/nitsの3段階で指摘→修正→返信のサイクルを回しました。
