# レビュー指摘修正スキル

PR のレビューコメントを取得し、指摘への修正・コミット・push・返信を一気通貫で行う。

## 使い方

```
/fix-review <PR番号>
```

## 実行手順

### 1. PR情報とレビューコメントの取得

```bash
# PR情報
gh pr view <PR番号> --json number,title,headRefName,baseRefName,url

# レビュー一覧（承認/変更要求/コメント状態）
gh api repos/{owner}/{repo}/pulls/<PR番号>/reviews

# インラインコメント（ファイル・行番号付き指摘）
gh api repos/{owner}/{repo}/pulls/<PR番号>/comments
```

### 2. 指摘の分類

コメントのプレフィックスに基づいて severity を分類する:

- `🔴` または `[must` を含む → **must**（必ず修正）
- `🟡` または `[imo` を含む → **imo**（修正推奨）
- `🟢` または `[nits` を含む → **nits**（軽微な提案）
- 上記に該当しない → **その他**

**対応不要の除外条件:**
- 既に「修正しました」等の返信があるスレッド
- bot による自動コメント

### 3. 修正方針の提示と承認

**必ずユーザーの承認を得てから修正に着手すること。**

以下のフォーマットで修正方針を提示する:

```
## 📋 レビュー指摘サマリー

### 🔴 must（N件）— 必ず修正
1. [ファイルパス:行番号] 指摘内容
   → 修正方針: 具体的な修正内容

### 🟡 imo（N件）— 修正推奨
1. [ファイルパス:行番号] 指摘内容
   → 修正方針: 具体的な修正内容

### 🟢 nits（N件）— 軽微な提案
1. [ファイルパス:行番号] 指摘内容
   → 修正方針: 具体的な修正内容

---
上記の方針で修正してよろしいですか？
（imo / nits のスキップも可能です）
```

ユーザーが承認するまで次のステップに進まない。
ユーザーが特定の指摘をスキップしたい場合はその指示に従う。

### 4. コード修正の実施

承認された方針に基づき、コードを修正する。

- Edit ツールを使ってファイルを修正する
- 修正後、型チェックを実行して問題がないことを確認する

```bash
npx tsc --noEmit
```

型チェックでエラーがあれば修正してから次に進む。

### 5. コミットとプッシュ

```bash
git add <修正したファイル>
git commit -m "fix: レビュー指摘を修正"
git push
```

コミットメッセージは修正内容に応じて適切に記述する。

### 6. レビューコメントへの返信

各指摘コメントに対して、コミットSHA付きで返信する。

```bash
gh api repos/{owner}/{repo}/pulls/<PR番号>/comments/<comment_id>/replies \
  --method POST \
  -f body="修正しました。 <commit_sha>

<修正内容の簡潔な説明>"
```

**返信に含める必須要素:**
- コミット SHA（短縮形）
- 何をどう修正したかの簡潔な説明

スキップした指摘には理由を返信する:
```
この指摘はスキップしました。理由: <ユーザーの判断理由>
```

### 7. 完了サマリーの表示

```
## ✅ レビュー修正完了

- 修正: N件（must: N / imo: N / nits: N）
- スキップ: N件
- コミット: <commit_sha>
- PR: <PR_URL>
```
